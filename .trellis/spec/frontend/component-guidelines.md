# Component Guidelines

> How components are built in this project.

日期：2026-06-17
执行者：Codex

---

## Core Rule

All base UI primitives must come from official shadcn-vue components generated by the official shadcn-vue CLI/registry.

The project must not hand-roll or maintain local replacements for primitives that shadcn-vue already provides, including buttons, badges, cards, inputs, selects, tables, dropdown menus, dialogs, toasts, popovers, tabs, and similar controls.

---

## Official Component Source

- Use `frontend/components.json` as the shadcn-vue configuration source.
- Add new primitives with the official CLI from the `frontend/` directory, for example:

```bash
pnpm dlx shadcn-vue@latest add button badge card input select table dropdown-menu sonner
```

- Generated primitives live under `frontend/src/components/ui/<component>/`.
- Business code imports primitives from directory exports such as `@/components/ui/button`, `@/components/ui/card`, and `@/components/ui/select`.
- Do not import or create single-file local wrappers such as `@/components/ui/Button.vue`, `BaseButton.vue`, `BaseCard.vue`, or `TableShell.vue`.

---

## Composition Boundary

Business components may compose official primitives, but they must not replace them.

Allowed:

- `frontend/src/components/market/SoftTable.vue` composing official `Card` and `Table` primitives.
- `frontend/src/components/market/StatCard.vue` composing official `Card`.
- Page components using official `Button`, `Badge`, `Input`, `Select`, `DropdownMenu`, and `Toaster` directly.

Forbidden:

- Reimplementing shadcn-vue primitive styling in local `Base*` components.
- Adding a compatibility wrapper only to preserve an old local API such as `<Select :options="...">`.
- Editing generated shadcn-vue source for ordinary product styling.
- Adding new fallback UI components when an official primitive exists.

---

## Generated Source Policy

Treat generated shadcn-vue files as vendor-style project source.

- Do not casually edit generated primitive files.
- If a generated primitive needs to be refreshed, rerun the official CLI and review the diff.
- Product-specific styling belongs in business components, page markup, Tailwind classes at call sites, or theme variables.
- If a local change to generated source is unavoidable, document the reason in this spec before making the change.

Current documented exception:

- `frontend/src/components/ui/sonner/Sonner.vue` uses `CheckCircledIcon`, `InfoCircledIcon`, `CrossCircledIcon`, and `ExclamationTriangleIcon` because the generated shadcn-vue template referenced icon names that are not exported by the installed official `@radix-icons/vue@1.0.0` package.
- `frontend/src/components/ui/badge/index.ts` keeps product semantic badge variants (`identity`, `verified`, `trust`, `capability`, `model`, `status`) in the generated badge variant map so every market tag shares one Aqua Console color contract instead of duplicating per-page badge classes.

---

## Toasts

Use the official shadcn-vue `sonner` component and `vue-sonner` API.

- Render `Toaster` once in `App.vue`.
- Use `toast(...)`, `toast.success(...)`, `toast.warning(...)`, or equivalent `vue-sonner` APIs in business code.
- Do not maintain a Pinia toast store or custom toast host for normal notifications.

---

## Responsive Dialogs

When a dialog is only meant for one breakpoint, gate the open state as well as the content classes. `DialogContent` can be hidden with responsive Tailwind classes, but the dialog overlay is rendered by the primitive separately and will still cover the page if the dialog is open.

Required pattern:

- Use the existing shadcn-vue `Dialog` primitive.
- Keep product-specific sheet styling at the call site.
- Before opening a mobile-only dialog from shared submit handlers, check the same breakpoint used by the CSS, for example `window.matchMedia('(max-width: 639px)').matches` for `sm:hidden`.
- Desktop failure states should use inline summaries, field errors, focus, and scroll targets instead of opening a hidden mobile sheet.

Avoid:

- Setting `open = true` unconditionally and relying on `sm:hidden` or `hidden` on `DialogContent` to suppress the desktop UI.
- Editing generated dialog primitives to solve one page's responsive behavior.

## Selects

Use the official composed select API:

```vue
<Select default-value="sub2api">
  <SelectTrigger class="w-full">
    <SelectValue placeholder="选择交付方式" />
  </SelectTrigger>
  <SelectContent>
    <SelectItem value="sub2api">Sub2API</SelectItem>
  </SelectContent>
</Select>
```

Do not recreate a native `<select>` wrapper with custom `options` props.

## Business State Components

Business components that look like controls must expose state to the owning page when the state changes displayed data.

Required pattern:

- Use `v-model` / `update:modelValue` for filter tabs, status tabs, segmented controls, and similar business controls.
- The page owns derived rows and side effects; the business component only renders choices and emits the selected value.
- A business component may keep local state only for display-only use, or as a compatibility default when `modelValue` is omitted.

Example:

```vue
<StatusTabs v-model="activeStatus" :items="['全部', '待处理', '已通过']" />
```

Avoid:

- Tabs that only change their own active button style while the table/list below stays unchanged.
- Duplicating row-filtering logic inside the tabs component.

---

## Review Checklist

- Did every primitive come from `frontend/src/components/ui/<component>/` generated by shadcn-vue?
- Did the change avoid local `Base*` UI components and compatibility wrappers?
- Are business components composing official primitives instead of reimplementing them?
- Are product-specific styles outside generated shadcn-vue files?
- Are toasts using official `sonner` instead of local store/host code?
- Do filter/status controls expose selected state to the page when rows or cards depend on that state?
