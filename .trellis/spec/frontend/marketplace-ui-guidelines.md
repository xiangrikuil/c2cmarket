# C2CMarket 全站 Marketplace UI 规范

日期：2026-07-14
执行者：Codex
来源：用户确认的《C2CMarket 全站 UI 风格基准》

## 1. 适用范围与优先级

本规范适用于所有面向用户、商户和管理员的前端页面，包括首页、订阅拼车、API 服务、求车需求、官网价格、订单与交易记录、发布资源、个人中心、商户工作台和管理台。

当历史截图、旧页面样式、可选主题或页面局部设计与本规范冲突时，以本规范为准。业务流程、权限、状态机、后端接口和产品边界仍以现有领域规范为准；视觉改造不得借机改变业务逻辑。

核心判断标准：页面首先应被识别为可浏览、比较和交易资源的 **C2C 市场平台**，而不是 AI 模型官网、AI SaaS 宣传页或科技概念站。

## 2. 产品表达契约

### 必须表达

- 市场属性：真实资源、服务、价格、库存、席位、额度、地区、卖家和信誉。
- 比较属性：相同类型的商品或服务使用稳定、对齐的字段层级。
- 状态属性：可申请、可接单、待确认、异常、关闭等状态必须明确。
- 行动路径：每个页面有清晰的下一步，主操作数量克制且语义稳定。
- 业务入口：订阅拼车、API 服务、求车需求、官网价格、订单记录和发布资源。

### 禁止表达

- 把“AI 能力”“智能”“未来科技”作为页面主视觉或首要卖点。
- 使用深蓝大面积渐变 Hero、3D 盾牌、3D 芯片、发光玻璃卡片、科技网格背景或大面积 AI 宣传插画。
- 用无业务含义的视觉装饰取代真实价格、库存、状态和交易信息。
- 在首页展示系统健康、运行指标等与用户交易决策无关的信息。
- 使用满员、过期、未审核或不可交易的资源伪装成推荐资源。

## 3. 设计令牌契约

全站应通过主题变量和公共组件消费这些颜色，不在页面中重复维护另一套近似色。

| 语义 | 固定值 | 使用范围 |
| --- | --- | --- |
| 品牌蓝 | `#2563EB` | 主按钮、主链接、选中态、关键数字 |
| 品牌蓝 Hover | `#1D4ED8` | 主按钮和主链接 Hover |
| 浅蓝背景 | `#EFF6FF` | 选中项、轻提示、Hero 浅底 |
| 页面背景 | `#F8FAFC` | App 内容区 |
| 卡片背景 | `#FFFFFF` | 卡片、表格、弹层 |
| 主文字 | `#0F172A` | 标题、价格、主要字段 |
| 次文字 | `#64748B` | 描述、辅助字段、时间 |
| 边框 | `#E2E8F0` | 卡片、表格、输入框分隔 |
| 成功 / 可用 | 绿色语义色 | 可申请、可售、已完成 |
| 等待 / 待确认 | 琥珀色语义色 | 等待对方、审核中、待确认 |
| 风险 / 异常 / 删除 | 红色语义色 | 错误、风险、破坏性操作 |
| 已关闭 / 历史 | 灰色语义色 | 失效、关闭、历史记录 |

模块辅助色只能用于小图标、Badge、浅色背景或细边框：订阅拼车使用靛蓝/紫蓝，API 服务使用青色，求车需求使用橙色，官网价格使用紫色。所有模块的主要行动按钮仍统一使用品牌蓝。

主题要求：可选主题不得改变品牌蓝主按钮、状态语义、信息层级和 Marketplace 页面骨架。后续统一改造应消除默认主题中的青色主按钮和过强装饰性主题效果。

## 4. 页面骨架与排版

### 布局

| 元素 | 桌面端契约 |
| --- | --- |
| 顶部导航 | `64px` 高 |
| 左侧导航 | 展开宽度 `184–200px`；收起态可保留紧凑宽度 |
| 内容区 | `#F8FAFC` 背景，左右内边距 `24px` |
| 主内容 | 使用可用桌面宽度；禁止退化为约 `320px` 的窄列 |
| 大卡片圆角 | `16px` |
| 普通卡片圆角 | `12px` |
| 页面主标题 | `28–32px` |
| 区块标题 | `18–20px` |
| 正文 | `14px` |
| 辅助信息 | `12px` |
| 间距系统 | 只优先使用 `8 / 12 / 16 / 24 / 32px` |

### 卡片和层级

- 卡片使用细边框和轻阴影；阴影只帮助区分层级，不制造悬浮科技感。
- 避免“卡片套卡片套卡片”。列表行、分组和表格优先使用分隔线、留白和浅背景建立层级。
- 页面标题、筛选、列表、详情和操作区应在全站保持稳定位置。
- 优先复用 `components/ui` 的 shadcn-vue 原语和 `components/market` 的业务展示组件，不创建第二套基础 UI。

## 5. 首页契约

首页顺序固定为：

1. Header。
2. 白色或极浅蓝色 Hero。
3. 细的动态公告滚动条，可内嵌于 Hero 底部或紧邻 Hero。
4. 四个业务入口卡：订阅拼车、API 服务、求车需求、官网价格。
5. 推荐资源：推荐拼车和推荐 API。
6. “为什么选择 C2CMarket”。
7. Footer（项目存在 Footer 时）。

### Hero

- 左侧：标题、说明、`浏览服务` 主按钮、`发布资源` 次按钮。
- 推荐标题：`AI 服务交易市场`。
- 推荐说明：`订阅拼车、API 服务、求车需求、官网价格查询`。
- 右侧：2–3 张真实资源浮层卡，只展示真实查询结果，不写死伪造库存和价格。
- 资源卡应突出产品名、价格和可用状态，例如“ChatGPT Plus 共享席位 / ¥38 月起”。示例文案只是格式说明，不得替代真实数据。
- Hero 使用白底或 `#EFF6FF` 浅底；禁止深蓝渐变、3D 盾牌、科技网格、发光玻璃和无业务含义统计墙。

### 首页限制

- 不增加右侧多层侧栏卡片，不让“快速开始”“平台状态”“帮助入口”等重复挤压主内容。
- 推荐区不得包含满员拼车、不可售 API、过期资源或未通过审核的条目。
- 平台边界和风险说明应清晰但克制，不能抢占资源比较的主要视觉层级。

## 6. 市场资源展示契约

### 推荐拼车

只有同时满足以下条件的车源才能进入推荐区：

```ts
type RecommendableCarpool = {
  approved: true
  status: '可上车'
  remainingSeats: number // > 0
  expiresAt: string // 当前时间之前未过期
}
```

字段优先级：产品名 → 价格 → 地区 → 账号/接入类型 → 原帖绑定 → 剩余席位 → 卖家/信誉。

### 推荐 API

只有同时满足以下条件的服务才能进入推荐区：

```ts
type RecommendableApiService = {
  approved: true
  publiclyOrderable: true
  availableQuotaUsd: number // > 0
}
```

字段优先级：服务名 → 售价 → 倍率 → 可售额度 → 最低订单 → 响应速度 → 商户信誉。

实际实现必须把这些展示条件映射到现有领域模型和权威查询字段，不能在页面新增一套推断状态，也不能为了凑推荐数量放宽资格。

### API 商家徽章

商家徽章只能由平台已有数据推导，不能由商家自行填写，也不能把普通身份或交易统计包装成平台背书：

- `优质商家`：`trustLevel >= 3`、近 30 天完成订单数 `completed30d >= 10`，且 `unresolvedDisputes === 0`。
- `快速响应`：服务当前允许公开下单，并且实际响应中位数已知且不高于 10 分钟；响应数据未知时不展示。
- `平台认证`：只有后端提供独立的认证状态、审核记录和有效期字段后才能展示，不能从实名认证、信用等级或成交量推断。
- 徽章数量最多 3 个；后续支持平台认证后，固定顺序为 `平台认证` → `优质商家` → `快速响应`。当前仅展示已有权威字段能够支撑的前两类经营表现徽章。

## 7. 公共组件契约

后续页面优化优先统一并复用以下结构：

- `AppShell`：64px 顶栏、184–200px 左侧栏、24px 内容边距。
- `PageTitle`：页面标题、简短说明和唯一主操作。
- `Card` / `SoftTable`：12px/16px 圆角、细边框、轻阴影。
- `Button`：主操作统一品牌蓝；模块辅助色不得创建新的主按钮 variant。
- `StatusBadge`：只按业务状态使用绿色、琥珀色、红色和灰色。
- `FilterBar`：常用筛选直接可见，高级筛选折叠，列表和表格字段对齐。
- `SkeletonBlock` / `SkeletonTable`、`EmptyState`、`ErrorState`：统一加载、空状态和失败恢复。
- 步骤表单：复用官方 Stepper、表单控件和现有校验，避免页面级自研控件。

## 8. 验收矩阵

| 场景 | 正确结果 | 不允许 |
| --- | --- | --- |
| 首页首屏 | 浅色 Hero + 真实资源卡 + 清晰双操作 | 深蓝渐变 + 3D 盾牌/芯片 + 科技网格 |
| 市场列表 | 价格、库存、状态、卖家可比较 | 大插画抢占空间、字段随卡片随机排列 |
| 模块入口 | 辅助色只在图标/浅底，主操作为品牌蓝 | 每个模块使用不同颜色主按钮 |
| 不可交易资源 | 显示明确禁用状态或不进入推荐 | 满员、过期、零额度资源作为热门推荐 |
| 桌面布局 | 充分利用内容宽度，边距 24px | 主内容被侧栏挤成约 320px 窄列 |
| 加载状态 | Skeleton，不显示真实感的零数据 | 加载中显示 0 库存、0 交易等假状态 |
| 空状态 | 说明原因并给出下一步 | 用宣传插画替代行动路径 |

## 9. Good / Base / Bad 案例

- Good：首页 Hero 左侧说明市场能力和两个行动，右侧展示三条真实可交易资源；下方公告条和业务入口帮助用户继续浏览。
- Good：API 服务卡在同一视觉顺序展示 `¥0.80 / $1`、倍率、可售额度、最低订单、响应速度和商户信誉。
- Base：推荐查询为空时展示真实空状态和前往对应市场的入口，不填充虚构卡片。
- Bad：首页使用深蓝大渐变、发光网格和 3D 盾牌作为视觉中心，真实资源只出现在第二屏。
- Bad：拼车卡只显示产品 Logo 和宣传语，却不显示价格、地区、剩余席位和卖家信誉。

## 10. 本地验证要求

每次产品页面视觉改造至少完成：

1. 运行相关单元/组件测试，确保筛选、状态和操作逻辑未改变。
2. 运行 `pnpm --dir frontend exec vue-tsc -b --pretty false`。
3. 运行 `pnpm --dir frontend exec vite build`，真实 API 模式要求沿用任务约定。
4. 使用真实浏览器验证桌面端，基准视口为 `1440×900`。
5. 首页和本次改动的代表页面必须留存截图并检查：市场感、信息层级、真实字段、主操作、横向溢出。
6. 涉及响应式布局时补充移动端检查；但设计和验收优先保证桌面端。

### 页面级审查问题

- 用户能否在首屏判断这是什么市场、有什么可交易资源、下一步去哪？
- 同类资源能否快速比较价格、库存/额度、状态和卖家信誉？
- 是否存在比真实业务信息更抢眼的装饰？
- 主按钮是否统一为品牌蓝，状态色是否表达正确语义？
- 是否复用了公共组件，且没有改动业务逻辑和后端接口？

## 11. Wrong vs Correct

### Wrong

```vue
<section class="bg-gradient-to-br from-blue-600 to-blue-950 text-white">
  <ThreeDShield />
  <h1>释放下一代 AI 能力</h1>
</section>
```

### Correct

```vue
<section class="rounded-2xl border border-border bg-blue-50 p-8">
  <div>
    <h1 class="text-3xl text-slate-900">AI 服务交易市场</h1>
    <p class="text-sm text-slate-500">订阅拼车、API 服务、求车需求、官网价格查询</p>
    <Button>浏览服务</Button>
    <Button variant="outline">发布资源</Button>
  </div>
  <RecommendedResourceCards :items="recommendableResources" />
</section>
```

示例表达的是布局与信息层级；实现时仍须复用项目现有组件、查询和领域字段。
