# Database Guidelines

> Database patterns and conventions for this project.

---

## Overview

Target database is PostgreSQL. Migration SQL under `backend/migrations/` is the database contract baseline for backend/API design. Runtime code may keep an in-memory implementation only for local tests or for a task that explicitly defers PostgreSQL because a prerequisite design, such as contact-value encryption, is not settled yet.

---

## Query Patterns

The process-level PostgreSQL runtime foundation is wired through `pgx/v5/pgxpool`. The current PostgreSQL repository slice covers users, auth identities, auth sessions, linux.do bindings, user permissions, idempotency, product categories, product plans, product plan policy history, official price leads, official price records, contact methods, contact sessions, contact session items, contact access logs, carpool listings, carpool cycle terms, carpool applications, carpool join confirmations, carpool memberships, carpool completion confirmations, API services, API service child tables, API purchase intents, API purchase intent contact access logs, API orders, API order events, API order payment-instruction access logs, user profile privacy fields, merchant profiles, announcements, announcement receipts, announcement audit logs, favorites, carpool reviews, reports, dispute cases, appeals, and dispute events.

- Preserve service-level contracts from `backend/internal/module/core` during the transition, and add or extend focused repository interfaces for the touched business domain. New durable storage work should not depend on one broad repository interface.
- Use transactions for multi-row state changes such as official price approval, carpool application acceptance, contact session creation, and idempotency completion.
- For durable create/action endpoints that produce multiple side effects, write the completed idempotency metadata or response cache inside the same transaction as the business rows. Official price approval is the first baseline pattern: lead update, price record, domain event, admin audit log, notification, and `idempotency_keys.status='completed'` commit together. Carpool application acceptance is the second baseline pattern: lock application/listing, check seats, create contact session and contact session items, update application, insert domain event and notification, and complete idempotency in one transaction. Carpool application cancel/withdraw is the paired reservation-ending pattern: lock application, validate actor/version/status, update application status/reason, revoke or expire the related contact session without clearing `contact_session_id`, write event/notification, and complete idempotency in one transaction. Carpool join confirmation is the third baseline pattern: lock application/listing, record the actor confirmation, create membership and increment `active_buyer_members` only when both sides have confirmed, write event/notification, and complete idempotency in one transaction. Carpool membership completion/exit is the fourth baseline pattern: lock membership/listing, record completion confirmations or ending reason, decrement `active_buyer_members` exactly once, revoke the membership application's contact session, write event/notification, and complete idempotency in one transaction. API purchase-intent creation and lifecycle actions are the fifth baseline pattern: lock the processing idempotency row, lock the public API service or intent row, create or update the intent, freeze buyer/owner contact method version IDs, write event/notification, and complete idempotency in one transaction. API order creation and lifecycle actions are the sixth baseline pattern: lock the processing idempotency row, lock the purchase intent or order row, create/update the API order, append `api_order_events`, insert payment-instruction access logs when reading private instructions, and complete idempotency in one transaction. Report, dispute, and appeal creation/actions are the seventh baseline pattern: lock the processing idempotency row, create or update the report/dispute/appeal row, append a `dispute_events` row, and complete idempotency in one transaction. API create responses contain `merchantContact.value`; API order responses can contain payment summaries or delivery notes. Idempotency rows for these sensitive responses must store resource identifiers and must not cache plaintext private values.
- Use row-level locking or unique constraints for idempotency and active-offer conflicts; do not rely on application-only duplicate checks.
- Nullable PostgreSQL response-cache fields must be scanned with explicit `COALESCE` or nullable Go types. A non-expired `processing` idempotency row has no response body, status, or resource id yet, and retrying that key must return `IDEMPOTENCY_IN_PROGRESS`, not an internal error. If the same user/route/key has the same request hash and `expires_at < now()`, PostgreSQL code may lock and take over the expired row for retry; the same key with a different request hash must still return `IDEMPOTENCY_KEY_REUSED`.
- Storage failures must surface as domain errors or Problem Details. Do not convert repository errors into empty lists or apparent success.
- Product catalog admin writes must preserve historical references by toggling `product_plans.active` instead of deleting rows. Policy-field changes on `product_plans` must lock the row, increment `policy_version`, update `policy_updated_at` / `policy_updated_by_user_id`, and insert `product_plan_policy_history` in the same transaction. Non-policy display/order/active changes must not append policy history.
- OAuth login writes identity data only. `auth_identities` stores provider and provider subject; `linux_do_bindings` stores public binding summary such as linux.do user id, username, trust level, avatar URL, and sync timestamps. Do not add provider access token, refresh token, OAuth code, raw userinfo JSON, password, cookie, or session fields to these tables.
- Native website accounts may store password credentials only in `user_password_credentials` as salted hashes. Current writes must use `argon2id_v1`; `sha256_salted_v1` is legacy verification-only and must be rehashed on successful login. Do not store plaintext passwords in `users`, `auth_identities`, `linux_do_bindings`, `auth_sessions`, logs, OpenAPI examples, seed docs, or frontend state. Migrations must not seed fixed admin password hashes; first-admin setup belongs to the explicit bootstrap environment path.
- OAuth login upserts `users`, `auth_identities`, and `linux_do_bindings` in one transaction. `auth_sessions` stores only hashed opaque session and CSRF tokens; the plaintext cookie/CSRF values are response-only secrets.
- Contact values must be encrypted before insertion into `contact_method_versions.value_ciphertext` and must have a keyed HMAC fingerprint. Store only ciphertext, nonce, masked value, key versions, and fingerprint; never store plaintext contact values in PostgreSQL.
- Contact session reads must lock/check the session, verify participant membership, reject expired/revoked windows before decryption, insert `contact_access_logs`, and return full values only in the authorized response.
- `contact_methods.current_version_id` must be constrained to a version with the same `contact_method_id` and `owner_user_id`; do not rely on service-only checks.
- `contact_session_items` must be constrained so `side=buyer` references the session buyer and `side=seller` references the session seller. The current schema uses a PostgreSQL constraint trigger for this.
- Carpool application uniqueness is database-backed with a partial unique index on `(carpool_listing_id, buyer_user_id)` for ongoing statuses. Service checks are useful for clearer errors, but the database constraint remains the authority.
- Carpool seat reservation is not a denormalized counter in this slice. Owner accept must lock the listing row, compute reserved seats from unexpired `accepted_reserved` applications, and fail when `buyer_seat_capacity - active_buyer_members - unexpired_reserved_seats < requested seats`. Both seat fields count buyer seats only and exclude the listing owner.
- `accepted_reserved` must have an absolute `reservation_expires_at`. Reads and state-changing actions must ignore expired reservations even if no scheduler has materialized them to `expired`.
- `accepted_reserved` must also have `join_confirmation_deadline`. Buyer/owner join confirmations must be actor-constrained by database trigger and deadline-constrained by service plus database checks. The first confirmation keeps the application reserved; the second confirmation moves it to `joined`.
- Application status values include `pending_owner`, `accepted_reserved`, `joined`, `rejected`, `cancelled_by_buyer`, `cancelled_by_owner`, and `expired`. `cancelled_by_owner` is used only when the owner withdraws an accepted reservation; pending owner denials remain `rejected`.
- `carpool_applications.contact_session_id` is a historical association. Constraints and triggers must allow non-null contact sessions for `accepted_reserved`, `joined`, `expired`, `cancelled_by_buyer`, and `cancelled_by_owner`; access control comes from `contact_sessions.status='open' AND ends_at > now()`, not from clearing the application pointer.
- Reservation-ending transitions must close contact sessions in the same transaction. Manual cancel/withdraw/leave/remove use `contact_sessions.status='revoked'` and `ends_at <= now()`. Reservation expiry uses `status='expired'` or otherwise guarantees reads no longer treat the session as open.
- `carpool_memberships` is the durable joined-buyer relationship. It must reference the exact `(application, listing, buyer, owner, product_plan)` tuple and can only be inserted for an application whose status is `joined` and `joined_at` is set. `active_buyer_members` is a listing cache that must be updated in the same transaction as membership creation.
- `carpool_cycle_terms` stores structured billing period, exit policy, and usage rules for a listing. It is part of the listing contract and should be written when a listing is created or updated, not left as an unrelated optional table.
- `carpool_completion_confirmations` records buyer/owner completion confirmation rows with actor constraints. Only the second confirmation may move membership to `completed`; buyer leave and owner remove must go through explicit `left` / `removed` transitions with a reason.
- `carpool_listings.owner_contact_method_id` records the owner-selected contact method; `carpool_applications.buyer_contact_method_id` records the buyer-selected contact method. Accept freezes the current versions from those stored selections and must not depend on a mutable default contact method or accept-body contact method.
- `carpool_applications.contact_session_id` must be one-to-one and participant-consistent with the contact session. Database constraints/triggers should enforce session buyer = application buyer and session seller = application owner.
- Carpool risk acknowledgement is business-scoped in `carpool_listing_policy_acknowledgements` and `carpool_application_policy_acknowledgements`. Do not replace it with a loose generic acknowledgement row unless the target business row can be constrained with an explicit foreign key and version snapshot.
- Carpool owner queues and owner membership lists are ownership views over `carpool_listings.owner_user_id` / `carpool_memberships.owner_user_id`; there is no separate merchant role or merchant-profile requirement for `/owner/carpool-applications` in this slice.
- API market service publishing uses `api_model_catalog`, `api_model_price_versions`, `api_services`, `api_service_access_modes`, `api_service_models`, `api_service_packages`, `api_services.accepting_orders`, `api_services.payment_window_minutes`, and `api_service_payment_options`. API purchase intents live in migration `000014_api_market_purchase_intents` and must stay separate from the service publishing migration.
- API service public visibility is database-readable from the shared orderable predicate: `review_status='approved'`, `publication_status='online'`, `moderation_status='clear'`, `accepting_orders=true`, `payment_window_minutes BETWEEN 3 AND 15`, and at least one enabled row in `api_service_payment_options`. Use the same predicate for public list/detail, search API-service and merchant projections, favorite target validation/listing, and purchase-intent creation.
- `api_services.owner_contact_method_id` must reference a contact method owned by the same `owner_user_id`; `merchant_profile_id` must reference a merchant profile owned by the same user when store alias mode is used.
- `api_service_models.distribution_system` duplicates the parent service distribution system through a composite foreign key so PostgreSQL can enforce `distribution_system <> 'sub2api' OR merchant_multiplier = 1.0000`.
- API service service-layer validation and database constraints both enforce the fixed Sub2API multiplier. Do not rely on frontend-only disabling or UI copy for this rule.
- API service tables store non-sensitive descriptions, access-mode notes, model snapshots, merchant-declared pricing, package descriptions, and private payment instructions only. Public reads may expose payment method labels but must not expose payment instructions, account numbers, wallet addresses, or payment-note text. API service tables must not store credential values, endpoint secrets, panel login materials, payment proof, or platform verification snapshots.
- `api_purchase_intents` stores immutable non-sensitive snapshots of the API service, requested amount fields, frozen buyer/owner contact method version IDs, and lifecycle timestamps. It must not store credential values, endpoint secrets, panel login materials, payment proof, quota ledger entries, platform verification snapshots, or plaintext contact values.
- API purchase intents must not create or reference `contact_sessions`; that table remains for carpool and development contact-window flows.
- API purchase-intent creation must reject services that are not public/orderable by the shared orderable predicate. Approved/online/clear but not accepting orders is still invisible to the public intent path.
- API purchase-intent cancel/close must reject once any `api_orders` row exists for that intent. After order creation, `api_orders` is the lifecycle source for buyer/seller actions.
- API purchase-intent amount validation treats `declared_max_usd_allowance_per_intent` as a per-intent discussion cap, not an inventory ledger. Do not add reservation, deduction, or fulfillment tables as side effects of creation.
- API purchase-intent direct contact disclosure must insert `api_purchase_intent_contact_access_logs` rows when the buyer views the merchant contact or the owner views the buyer contact. The log stores `api_purchase_intent_id`, `viewer_user_id`, `viewed_contact_owner_side`, `request_id`, and `accessed_at` only; it must not store plaintext contact values, masked contact values, credentials, payment evidence, or fulfillment data.
- API orders live in `api_orders`; state history lives in append-only `api_order_events`; explicit payment-instruction reads live in `api_order_payment_instruction_access_logs`. These tables do not create payment ledgers, wallets, escrow, refunds, API proxying, or credential delivery storage.
- `api_orders.api_purchase_intent_id` must have a full unique index, not a partial active-status index. The product contract is one order ever per purchase intent, including cancelled, timeout-cancelled, and completed orders. Runtime code must map PostgreSQL unique violation `23505` on `ux_api_orders_intent` to `409 API_PURCHASE_INTENT_HAS_ORDER` without leaking the raw constraint name or returning 500.
- API order disputes must create a `dispute_cases` row with `target_type='api_order'` and write the resulting ID into `api_orders.dispute_case_id` in the same transaction as the API order idempotency completion. The linked dispute case is an admin processing queue item only, not a refund, compensation, escrow, guarantee, or forced fulfillment mechanism.
- Payment timeout materialization must update `api_orders` from `pending_payment` to `cancelled/payment_timeout` atomically and write at most one `api_order.payment_timeout_cancelled` event per order.
- `api_order_payment_instruction_access_logs` stores only `api_order_id`, `buyer_user_id`, `request_id`, and `accessed_at`. It must not store payment instructions, account numbers, wallet addresses, screenshots, payment proofs, or delivery notes.
- User profile storage extends `users` with editable public fields and `privacy_settings` JSON. Public profile reads must apply privacy flags at query/DTO mapping time and must not join contact method values into public responses.
- Merchant profiles are owner-scoped and slug-addressable. Store-alias API services must reference a merchant profile owned by the same user through database constraints where available, and public merchant profile reads must omit owner user ID and contact values.
- Announcements live in `announcements`; per-user receipt state lives in `announcement_receipts`; admin audit events live in `announcement_audit_logs`. User-visible announcement reads should combine announcement status/time/channel predicates with the current user's receipt state.
- Announcement dismissal is per user and per announcement version. It must update only `announcement_receipts.dismissed_at`; do not mutate the source announcement to hide it from other users.
- Announcement queries that join `announcement_receipts` and need to lock only the source announcement row must use `FOR UPDATE OF a` or the relevant announcement alias. Bare `FOR UPDATE` on a `LEFT JOIN` can try to lock the nullable receipt side and fail in PostgreSQL.
- Favorites live in `favorites` with `user_id`, `target_type`, `target_id`, `created_at`, and a unique `(user_id, target_type, target_id)` constraint.
- Favorite target types are limited to `carpool` and `api_service`. Because `target_id` is polymorphic, PostgreSQL cannot express one direct foreign key for all targets; repository/service code must validate public target visibility before insert and list queries must join target tables with the public predicates.
- Favorite list queries must return only currently public-visible targets: active carpool listings and API services matching the shared orderable predicate.
- Favorites must not create notifications, contact sessions, payment rows, fulfillment rows, or credential snapshots. They are per-user markers only.
- Carpool reviews live in `carpool_reviews`. The only durable source type is `carpool_membership`; reviewer role is `buyer`, reviewee role is `owner`, and `source_id` references `carpool_memberships(id)`.
- `carpool_reviews` uses a unique `(source_type, source_id, reviewer_user_id)` constraint so repeated review submission updates the original review.
- A constraint trigger must verify the source membership is `completed`, the reviewer is `carpool_memberships.buyer_user_id`, and the reviewee is `carpool_memberships.owner_user_id`. Do not rely only on HTTP handler checks for this actor/source contract.
- Review center queries should build rows from completed buyer memberships and left join reviews so the UI can show both `reviewable` and `reviewed` states from one backend source.
- Public review queries should join by reviewee username and return only public-safe fields: service title/type, rating, tags, note, date, and verified flag. They must not expose reviewer IDs, contact values, contact method IDs, private membership internals, payment fields, or admin audit fields.
- Reviews must not create notifications, contact sessions, payment rows, fulfillment rows, dispute rows, compensation rows, or credential snapshots. Disputes and reports are separate modules.
- Reports live in `reports`; disputes live in `dispute_cases`; appeals live in `appeals`; append-only audit/history entries live in `dispute_events`.
- Report target types are limited to `contact_snapshot`, `public_user`, `carpool_membership`, `api_purchase_intent`, and `api_order`. `target_id` is text because public users and contact snapshots are not always UUID-backed domain rows; service code must validate required fields per target type.
- Report, dispute, and appeal admin actions must increment `version`, set handler fields, append `dispute_events`, and complete idempotency in the same transaction.
- Public dispute list/profile stats must read only from `dispute_cases` and return sanitized `public_summary` / `public_result` projections. Do not join raw report descriptions, appeal statements, admin reasons, reporter IDs, admin IDs, contact values, evidence, or internal notes into public responses.
- Reports/disputes/appeals must not create payment rows, refund rows, compensation rows, escrow rows, guarantee rows, fulfillment rows, contact sessions, credential snapshots, external tickets, emails, webhooks, file uploads, or automatic penalty rows.
- Notifications live in `notifications` and are a site inbox over durable business events. Inbox reads must filter by `user_id`, sort by `created_at DESC`, derive unread state from `read_at IS NULL`, and update only the current user's rows for single-read and read-all actions. Do not add external push, email, SMS, webhook, ticket, contact-value, payment, escrow, fulfillment, or credential-delivery side effects to notification reads.
- Search is read-only and aggregates existing public rows with the same predicates as public list endpoints. It may use `ILIKE` over public-safe fields, but must not query or return encrypted contact values, contact method IDs, admin internals, store-alias owner usernames, report raw text, or credential-looking data. Store-alias API service search may match and display public merchant display names only.
- Search indexes may use PostgreSQL `pg_trgm` with GIN trigram indexes on public-safe expression fields. These indexes are performance support only and must not broaden search visibility predicates or add a separate search table without a documented task.

## Migrations

- Migration files live in `backend/migrations/`.
- Use numbered `golang-migrate` names such as `000001_extensions_and_identity.up.sql` and matching `.down.sql` files.
- SQL must be PostgreSQL-oriented: `uuid`, `gen_random_uuid()`, `numeric`, `timestamptz`, `jsonb`, explicit `CHECK` constraints, foreign keys, and named/meaningful indexes.
- Extension migrations may enable `pgcrypto` and `pg_trgm`. Trigram indexes should use named GIN indexes over lowercased public-safe text expressions and must have matching down migrations.
- Public IDs are UUIDs; do not expose serial IDs in API contracts.
- Monetary amounts, FX rates, and multipliers use `numeric`, never `float`.
- Local Docker PostgreSQL uses the root `compose.yaml` `migrate` one-shot service. Run `docker compose --profile migrate run --rm migrate` after PostgreSQL is healthy.
- Do not rely on `/docker-entrypoint-initdb.d/` for application schema upgrades; it only runs for empty database volumes.
- PostgreSQL 18 Docker images must mount persistent storage at `/var/lib/postgresql`, not `/var/lib/postgresql/data`, because the image stores versioned cluster data below that parent directory.

## Docker Runtime

- Root `compose.yaml` owns local PostgreSQL and the backend image definition.
- Use `postgres:18-alpine` for the local database service unless a task explicitly changes the supported PostgreSQL line.
- Local database defaults are documented in `.env.example`; real `.env` files stay ignored.
- `DATABASE_URL` is read during `cmd/api` startup and wired through `internal/app`. When set, the process creates a `pgxpool` connection via `internal/database`, `/readyz` checks `schema_migrations`, and the current supported PostgreSQL slice writes users, auth identities, linux.do bindings, sessions, idempotency, official price data, contact-window data, carpool data, API services, API purchase intents, API purchase intent contact access logs, API orders, API order events, API order payment-instruction access logs, profile/merchant data, announcements, demands, favorites, reviews, reports, disputes, appeals, dispute events, and notifications to the database. Search reads those existing public rows and must not introduce a search table unless a future task documents a real performance need.
- `APP_ENV=production` requires `DATABASE_URL`, `FRONTEND_ORIGIN` or `ALLOWED_ORIGINS`, `CONTACT_ENCRYPTION_KEY`, `CONTACT_FINGERPRINT_KEY`, `CONTACT_KEY_VERSION`, `OAUTH_PROVIDER_MODE=oauth2`, `OAUTH_CLIENT_ID`, `OAUTH_CLIENT_SECRET`, `OAUTH_AUTHORIZE_URL`, `OAUTH_TOKEN_URL`, `OAUTH_USERINFO_URL`, `OAUTH_REDIRECT_URL`, `EMAIL_PROVIDER=aliyun_directmail`, `SMTP_HOST`, `SMTP_PORT=465`, `SMTP_USERNAME`, `SMTP_PASSWORD`, `MAIL_FROM_ADDRESS`, and `MAIL_FROM_NAME`, and rejects `ENABLE_DEV_AUTH=true`.
- Production email uses Aliyun DirectMail SMTP, not AccessKey or DirectMail API SDK. SMTP password values must remain environment-only and never be written into logs, persisted rows, docs, or AppError details.
- `email_verification_codes` supports login-bound `bind_email` challenges and public `email_registration` challenges. `email_registration` rows have `user_id IS NULL`; confirm must lock and consume the challenge, create the verified-email `users` row, and create `auth_sessions` in the same transaction. Store only `code_hash`, never plaintext verification codes.

## Naming Conventions

- Tables and columns use `snake_case`.
- JSON API fields use `camelCase`; conversion belongs at the HTTP/service boundary.
- Status fields use stable lowercase snake_case values.
- Unique indexes should use a `ux_` prefix plus the table and purpose.
- Contact values are modeled as versioned contact method rows. Runtime code must freeze selected versions into contact sessions instead of reading a user's latest contact value.
- Historical frozen contact method versions remain available for authorized dispute/audit reads after a user-facing contact method is retired. Do not physically destroy historical ciphertext without adding explicit `destroyed_at` semantics, key-rotation/destruction runbook steps, and tests for affected contact-session/API-intent reads.

## Common Mistakes

- Do not accept client-supplied authority fields such as `fx_rate`, `normalized_monthly_cny`, `fingerprint`, or `offer_key` for public low-price lead submissions. Those are service/admin-derived fields.
- Do not model contact sessions as pointers to mutable contact methods only; store/freeze contact method version IDs.
- Do not write broad seed down migrations that delete entire catalog tables. Seed rollback may delete only known seed UUIDs, and existing business FK references should block rollback rather than force data removal.
- Do not implement idempotency as a cache-only concern once PostgreSQL runtime exists. It needs a durable `processing/completed` record, a unique `(user_id, route_key, idempotency_key)` constraint, row locking, and transaction-coupled completion for actions that write business side effects.
- Do not treat expired and non-expired `processing` idempotency rows the same. Non-expired rows return `IDEMPOTENCY_IN_PROGRESS`; expired rows with the same request hash can be taken over for retry; expired rows with a different request hash still return `IDEMPOTENCY_KEY_REUSED`.
- Do not hard-code Plus/Pro or Business behavior in Go queries or handlers. Carpool creation and application rules must resolve product plan policy fields from the catalog.
- Do not update carpool application status to `accepted_reserved` before contact session rows and idempotency completion are ready to commit in the same transaction.
- Do not update a carpool application to `joined` without inserting the corresponding `carpool_memberships` row, incrementing `active_buyer_members`, and completing the idempotency response cache in the same transaction.
- Do not update a membership to `completed`, `left`, or `removed` without decrementing `active_buyer_members`, writing event/notification, and completing the idempotency response cache in the same transaction.
- Do not treat `carpool_applications.contact_session_id` as an active permission bit. Cancelling an application or ending a membership must leave the historical pointer intact and close the `contact_sessions` row instead.
- Do not model `/owner/*` carpool data through a merchant profile join. A regular user can be buyer, listing owner, and API merchant; resource ownership comes from the target row's owner fields.
- Do not expose `owner_contact_method_id`, admin review fields, moderation reasons, or `merchant_note` through public API service read DTOs. Those fields are owner/admin view data.
- Do not create API purchase-intent rows without frozen buyer/owner contact method versions, domain event, notification, and completed idempotency metadata in the same transaction.
- Do not expose API purchase-intent contact values from public lists, buyer/owner lists, admin endpoints, events, notifications, audit logs, regular logs, or idempotency response bodies. Full frozen values are allowed only in successful create, buyer detail, and owner detail responses with `Cache-Control: no-store`.
- Do not use `contact_sessions` or process logs as the audit mechanism for API purchase intent direct contact disclosure. Use `api_purchase_intent_contact_access_logs` and store identifiers/side metadata only.
- Do not model API order uniqueness as "active order only." The database must reject any second order for the same purchase intent even after cancellation, payment timeout, or completion.
- Do not store API order payment instructions in public service DTOs, access logs, events, notifications, regular logs, or cached idempotency response bodies. The order row may keep the frozen instruction snapshot, and the explicit buyer read endpoint returns it with `Cache-Control: private, no-store`.
- Do not store or auto-deliver API keys, Sub2API keys, passwords, tokens, sessions, cookies, recovery codes, endpoint secrets, subscription links, proxy-node URLs, encoded subscription URLs, nested subscription URLs, or attachment payloads in `api_orders.delivery_note`, events, or notification bodies. Treat the user-facing meaning of `delivery_note` as an off-platform delivery prompt, not platform delivery.
- Do not expose profile contact values, contact method IDs, merchant-profile owner IDs, or hidden owner mappings through public user/merchant profile routes.
- Do not model announcement read/dismiss state as fields on `announcements`; it is per-user receipt state and belongs in `announcement_receipts`.
- Do not create a favorite row before checking the target is public-visible. `favorites.target_id` has no direct polymorphic FK, so missing this check can persist favorites to private or non-existent targets.
- Do not create a review row for a non-completed membership or an API purchase intent. Current API purchase intents do not have a platform-confirmed completed state, so they are not review sources.
- Do not expose review rows as dispute, refund, guarantee, or fulfillment evidence. Reviews are public experience notes; dispute/report workflows must use their own durable tables and state machines.
- Do not build public dispute summaries from raw report descriptions, appeal statements, admin reasons, evidence URLs, contact values, or internal notes. Public dispute summaries must come from the explicit public fields on `dispute_cases`.
- Do not model reports/disputes/appeals as refund, compensation, escrow, guarantee, fulfillment, external ticket, or automatic penalty systems. They are manual risk records and public-safe summaries only.
- Do not extend `auth_identities`, `linux_do_bindings`, or `auth_sessions` to persist OAuth access tokens, refresh tokens, callback codes, plaintext passwords, cookies, raw provider payloads, or credential-bearing logs. The database contract stores identity linkage and hashed local session secrets only; native password hashes belong only in `user_password_credentials`.
