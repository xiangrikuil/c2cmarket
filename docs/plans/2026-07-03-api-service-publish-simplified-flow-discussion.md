# API 服务发布页简化流程讨论稿

日期：2026-07-03

用途：给 GPT 做产品与交互评审。本文不要求直接实现，先比较当前发布逻辑和建议的简化逻辑，重点评估是否能让商户更快发布 API 额度服务，同时不破坏 C2CMarket 的产品边界。

## 背景

当前发布页已经做过一轮压缩：`NewAPI Proxy` 不再作为顶级分发系统，归入 `其他 API 接入`；接入方式和用量可见性也已经从主表单中弱化，由系统使用保守默认值。

但现在页面仍然偏像审核后台表单。商户的真实意图通常是：

1. 我要出售多少额度，按多少钱卖。
2. 我要出售哪些模型。
3. 我要补充服务说明、用量核对和售后口径。

现有页面要求商户同时理解分发系统、模型大类、模型倍率、图像能力、最大额度、最低意向、最高意向、有效期、短句、承诺类型、承诺天数、适用范围、补偿方式、不适用情形、买家须知、展示身份等概念。首用成本仍然过高。

## 当前发布逻辑

### 当前主流程

```text
发布 API 服务

1. 分发系统
   - Sub2API
   - 其他 API 接入
   - 联动计费方式

2. 模型大类
   - GPT
   - Claude
   - 其他

3. 模型与计费
   - 选择模型
   - 已选模型明细表
   - Sub2API 固定文本倍率 1.00x
   - 其他 API 接入可配置默认倍率

4. 价格、库存与有效期
   - 最大可售美元额度
   - 最低意向金额
   - 单笔最高意向
   - 站外确认后有效期
   - 服务补充短句
   - 其他 API 接入还可能填写计费说明与用量核对方式

5. 图像能力
   - 仅 GPT + Sub2API + 已选模型支持图像能力时展示

6. 交易说明与承诺
   - 不作承诺
   - 上游退款跟随
   - 商户承诺
   - 承诺天数
   - 适用范围
   - 补偿方式
   - 不适用情形
   - 买家须知长文本

7. 提交前展示设置
   - 是否隐藏社区身份
   - 商家展示名
   - 预览
   - 发布 API 服务
```

### 当前表单默认值

```ts
distributionSystem = "sub2api"
billingMode = "metered_credit"
deliveryModes = ["api_key_endpoint"]
usageVisibility = "merchant_confirmed"
defaultMultiplier = 1.00
cnyPerUsdCredit = 0.8
availableCreditUsd = 500
minimumPurchaseCny = 20
maximumPurchaseCny = 300
validity = 30 days
warranty.mode = "merchant_warranty"
merchantNote = "建议首次提交 ¥20 意向测试..."
```

### 当前必填校验

当前发布前至少需要满足：

1. 展示身份有效；如果使用商家展示名，展示名必须 2-20 字且不能包含联系方式、链接、linux.do 用户名或误导词。
2. 必须选择模型大类。
3. 必须至少选择一个模型。
4. 已选模型必须属于当前模型大类。
5. Sub2API 必须填写合法的美元额度售价和最大可售美元额度。
6. 非固定套餐必须填写最低意向金额。
7. 固定套餐必须配置套餐。
8. 有效期必须合法。
9. 如果开启图像能力，必须符合 GPT + Sub2API + 图像模型条件。
10. 发布前不能选择 `不作承诺`，`上游退款跟随` 和 `商户承诺` 又各自有额外字段。
11. 买家须知不能为空，最多 800 字。
12. 所有文本都不能包含 API Key、Sub2API key、endpoint 密钥、token、密码或付款码。

### 当前后端映射

真实后端 adapter 仍按现有 DTO 提交。重要点是：真实后端并不需要前端的三段式售后承诺结构；当前 adapter 已经固定发送：

```ts
merchantSupportNote = "平台不担保、不代赔；双方站外确认。"
publicAccessNote = distributionSystemNote
merchantNote = merchantNote
```

也就是说，当前“承诺方式 / 承诺天数 / 适用范围 / 补偿方式 / 不适用情形”主要是前端发布体验和 mock 展示逻辑，不是后端必须保持的结构。

## 当前问题

### 1. 商户要理解太多平台概念

商户想卖额度，但页面要求先理解分发系统、计费模式、用量可见、售后模式、有效期、展示身份。对新用户来说，很多字段并不能帮助他们完成发布，只是在拖慢决策。

### 2. 信息输入重复

现在类似信息分散在多个地方：

- `distributionSystemNote`：分发系统说明。
- `manualBillingNote`：计费说明与用量核对。
- `warranty.coverage / compensation / exclusions / refundNote`：售后说明。
- `merchantNote`：买家须知。

这些最终都在表达“买家提交意向前应该知道什么”，完全可以合成一个备注长文本框。

### 3. 承诺三选一不符合快捷发布

`不作承诺 / 上游退款跟随 / 商户承诺` 看似严谨，但会让商户以为平台要求明确售后责任模型。C2CMarket 的边界是撮合和站外确认，不担保、不代赔。更合适的方式是平台固定展示边界声明，商户在备注里自由说明自己的售后口径。

### 4. 固定套餐让首屏复杂化

固定套餐需要名称、价格、有效期、说明、库存。它适合高级商户，但不适合作为默认发布路径。MVP 主流程应该只做“按美元额度售卖”。

## 修改后推荐逻辑

推荐把发布页改成“快捷发布 API 额度”，主流程只保留三个输入区。

### 1. 出售额度

用户只看到两个核心字段：

```text
每 $1 美元额度售价：¥____
可售美元额度：$____
```

默认隐藏字段：

```ts
distributionSystem = "sub2api"
billingMode = "metered_credit"
minimumPurchaseCny = 20
maximumPurchaseCny = 300
validity = 30 days
deliveryModes = ["api_key_endpoint"]
usageVisibility = "merchant_confirmed"
```

最低意向金额、单笔最高意向和有效期可以先不放主流程。若必须允许调整，放到“高级设置”折叠区，默认不展开。

### 2. 出售模型

保留模型大类和模型多选，但文案简化：

```text
出售模型
[ GPT ] [ Claude ] [ 其他 ]

搜索模型：__________

已选择 2 个模型：GPT-5 mini / GPT-5.1
```

模型卡片仍然在原位置高亮选中。下方“已选模型明细表”可以继续保留为折叠高级区，或者只在需要调整倍率时展开。

默认逻辑：

```ts
Sub2API 文本倍率固定 1.00x
图像能力配置默认关闭
如需图像能力，后续高级设置再打开
```

### 3. 备注信息

用一个长文本框合并所有说明：

```text
备注信息
请说明接入方式、用量核对、限速规则、可用时间和售后口径。
不要填写 API Key、token、密码、Session、Cookie、付款码或面板凭据。

[____________________________________________]
[____________________________________________]
[____________________________________________]
```

可以提供快捷 chips，但 chips 只是插入文本，不增加结构化字段：

```text
+ 建议首次提交 ¥20 意向测试
+ 提交意向后站外确认接入细节
+ 用量由商户说明，买家自行核对
+ 高峰期响应可能变慢
+ 部分模型可能临时维护
+ 滥用或高并发压测不在服务范围内
+ 平台不担保、不代赔
```

发布校验只要求备注不为空、长度不超过 800 字、不含敏感凭据。

### 4. 提交前设置

展示身份保留，但放在底部，不打断主流程：

```text
[x] 不公开社区身份，仅展示商家展示名
商家展示名：__________
```

它是展示偏好，不是服务配置。默认可继续使用 `store_alias`，降低商户暴露顾虑。

## 字段映射建议

### 用户填写字段

| UI 字段 | 写入字段 |
| --- | --- |
| 每 $1 美元额度售价 | `cnyPerUsdCredit` |
| 可售美元额度 | `availableCreditUsd` |
| 模型大类 | `providerCategory` |
| 已选模型 | `selectedModels` |
| 备注信息 | `merchantNote` |
| 商家展示名 | `merchantDisplayName` |
| 是否隐藏社区身份 | `merchantIdentityMode` |

### 系统默认字段

| 字段 | 默认值 | 说明 |
| --- | --- | --- |
| `distributionSystem` | `sub2api` | 默认按 API 额度服务处理。 |
| `billingMode` | `metered_credit` | 主流程只做按美元额度售卖。 |
| `deliveryModes` | `["api_key_endpoint"]` | 页面不展示接入方式选择，备注里说明。 |
| `usageVisibility` | `merchant_confirmed` | 平台不实时核验余额。 |
| `defaultMultiplier` | `1.00` | Sub2API 固定倍率。 |
| `minimumPurchaseCny` | `20` | 默认最低意向金额。 |
| `maximumPurchaseCny` | `300` | 默认单笔最高意向，可后续高级设置。 |
| `validity` | `30 days` | 默认站外确认后 30 天有效。 |
| `warranty.mode` | `no_warranty` 或删除前端硬校验 | 售后口径写入备注，不再强制三选一。 |
| `merchantSupportNote` | `平台不担保、不代赔；双方站外确认。` | 真实后端 adapter 已经固定设置。 |

## 实现方案

### 方案 A：只简化 UI，保持 DTO 兼容（推荐）

改动范围：

1. 删除或停用 `DistributionBillingSection` 中的分发系统主选择，默认 `sub2api`。
2. 把 `PriceInventorySection` 改成 `QuotaSaleSection`，只显示 `cnyPerUsdCredit` 和 `availableCreditUsd`。
3. 保留 `ProviderCategorySelector` 和 `ModelMultiSelect`。
4. 删除 `WarrantySection` 的三选一承诺 UI，改成 `MerchantNoteSection`。
5. `ApiServicePublishPage.vue` 中的校验改为：
   - 有合法商家展示名。
   - 有合法 `cnyPerUsdCredit`。
   - 有合法 `availableCreditUsd`。
   - 至少选择一个模型。
   - `merchantNote` 非空且不包含敏感凭据。
6. 提交 payload 仍按现有结构补齐默认值，不改后端。

优点：

- 用户最快能发布。
- 不动 Go 后端、migration、OpenAPI。
- 风险低，适合现在直接落地。

缺点：

- 固定套餐、其他 API 接入、高级售后承诺暂时不在主流程里。
- 如果未来要精细化交易规则，需要再加高级设置。

### 方案 B：保留高级模式

主流程仍按方案 A 简化，但增加一个折叠区：

```text
[高级设置]
- 最低意向金额
- 单笔最高意向
- 有效期
- 图像能力
- 模型倍率
- 固定套餐
- 其他 API 接入
```

优点：

- 兼顾高级商户。
- 以后扩展较平滑。

缺点：

- 即使折叠，也会让页面看起来“还有很多东西”。
- 首用用户可能仍然被高级入口干扰。

### 方案 C：完全改后端契约

新增一个更贴近快捷发布的后端请求：

```ts
CreateApiQuotaServiceRequest = {
  cnyPerUsdCredit: string
  availableCreditUsd: string
  selectedModelIds: string[]
  merchantNote: string
  merchantIdentityMode: "public_profile" | "store_alias"
  merchantDisplayName?: string
}
```

优点：

- 长期契约最干净。

缺点：

- 需要改 Go handler、service、OpenAPI、adapter、测试。
- 当前阶段不值得，容易扩大任务范围。

## 推荐结论

推荐先做方案 A：UI 快捷发布，DTO 兼容不动。

新的主路径应该是：

```text
出售额度 -> 出售模型 -> 备注信息 -> 发布
```

不建议继续保留“交易说明与承诺”三选一。平台统一展示“不担保、不代赔；双方站外确认”，商户自己的补偿、限速、退款、维护、不可用处理都写到备注里。

固定套餐和其他 API 接入可以先不进入主流程。若确实要保留，放到后续“高级设置”而不是默认展示。

## 需要 GPT 重点评审的问题

1. 只保留 `出售额度 / 出售模型 / 备注信息` 三段，是否足够覆盖商户发布 API 额度服务的最低必要信息？
2. 删除“承诺三选一”，改为平台固定边界声明 + 商户备注，是否会降低买家理解或信任？
3. `minimumPurchaseCny=20`、`maximumPurchaseCny=300`、`validity=30 days` 作为默认值是否合理？
4. 固定套餐是否应该完全移出主流程，还是作为折叠高级设置保留？
5. 是否还需要保留 `其他 API 接入` 发布能力，还是现在先默认只支持 Sub2API 风格的美元额度售卖？
6. 备注框是否需要结构化提示模板，还是纯长文本就足够？
