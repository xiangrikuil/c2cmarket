# API 服务发布页选中逻辑与表单压缩评审稿

日期：2026-07-02

用途：给 GPT 做产品与交互评审。本文不要求直接实现，先评审选中逻辑、字段删减和风险。

## 背景

当前“发布 API 服务”页更像审核后台表单，字段完整但首用成本高。主要问题有三类：

1. 选择模型后，用户的注意力容易被带到下方明细表；第一次使用时不一定知道“已经选中”。
2. 页面过长，发布 API 服务需要滚动很多次，决策路径不够短。
3. “对外展示身份”“接入方式”“用量与余额查看”这类字段混在主流程里，增加理解负担，其中后两项也未必需要由商户主动配置。

我的建议是：模型选择保留在原地高亮；发布主流程压缩；展示身份移动到最后；接入方式和用量查看从主表单移除，由系统保守默认或由商户在备注里说明。

## 推荐主流程

发布页保留这些主步骤：

1. 分发系统
2. 模型大类与模型选择
3. 价格、库存、最低意向、有效期
4. 商户承诺与买家须知
5. 提交前展示设置

图像能力可以继续作为模型相关的附加配置，但只在 GPT + Sub2API + 已选支持图像能力模型时出现。

2. 已选状态必须比 hover 状态更明显。
3. 点击后不滚动页面。
4. 下方明细表不是主要反馈，只是编辑区。

## 模型选中逻辑

### 目标

用户点击模型卡片后，反馈必须发生在原位置：卡片保持可见，并立即变成高亮选中态。不要依赖下面的“已选模型价格表”来告诉用户选中了什么。

### 状态来源

沿用现有表单结构：

```ts
form.selectedModels: Array<{
  modelId: string
  multiplierOverride: number | null
  enabled: boolean
}>
```

新增或派生一个本地集合：

```ts
selectedModelIds = new Set(
  form.selectedModels
    .filter(item => item.enabled)
    .map(item => item.modelId)
)
```

不需要改后端 payload，也不需要新增业务模型。

### 卡片展示规则

模型列表不再过滤掉已选模型。所有符合当前模型大类和搜索条件的模型都留在列表里。

卡片分三种状态：

| 状态 | 视觉 | 文案 |
|---|---|---|
| 未选中 | 普通边框、白底或默认背景 | 选择 |
| 已选中 | 主色边框、浅主色背景、轻微 ring、高亮角标 | 已选择 |
| 不可选 | 一般不显示；如果保留则置灰 | 不可用 |

已选中卡片建议放一个 `Check` 图标或明确角标。视觉上要比 hover 更强，用户不用看下方表格也能确认。

### 点击行为

推荐使用“点击切换”：

1. 点击未选中模型：加入 `form.selectedModels`。
2. 点击已选中模型，且当前已选数量大于 1：从 `form.selectedModels` 移除。
3. 点击已选中模型，且它是唯一已选模型：不移除，提示“至少保留一个模型”。
4. 点击后不自动滚动页面，不跳转到下方明细。

这样多选成本低，也不会让用户误删到空状态。

### 已选模型明细表

下方“已选模型价格表”可以保留，但定位改成明细编辑区，不作为主要选择反馈。

建议在模型列表顶部增加一行轻量摘要：

```text
已选择 3 个模型：GPT-5 mini / GPT-5.1 / GPT-4.1
```

如果已选模型较多，展示前 3 个，后面显示 `+N`。这比让用户滚动到表格里确认更直接。

### 搜索规则

搜索时，已选模型仍然参与搜索结果。如果已选模型匹配关键词，仍展示为高亮状态。

可选增强：把已选模型固定在当前搜索结果顶部，下面再展示未选模型。但这不是 MVP 必需项。

### 切换模型大类

保持现有的保护逻辑：如果切换模型大类会清空当前已选模型，需要二次确认。

推荐确认文案：

```text
切换到 Claude 会清空当前已选 GPT 模型。是否继续？
```

确认后执行：

1. 更新 `providerCategory`
2. 清空不属于新大类的 `selectedModels`
3. 如果新大类有可用模型，可以自动选择第一个模型；也可以保持空状态并要求用户选择

我更倾向于自动选择第一个模型，因为发布页默认要服务首用者，但需要让这个自动选择也有明显高亮。

## 表单压缩建议

### 对外展示身份

当前放在第一步会过早打断发布任务。它不是服务能力本身，而是提交前展示偏好。

建议移动到提交区附近，做成一个小设置：

```text
[x] 不公开社区身份，仅展示商家展示名
```

勾选后显示商家展示名输入框。默认可以保持当前逻辑：使用商家展示名，隐藏 linux.do 用户名和个人主页。

### 接入方式

建议从主发布表单移除。

理由：

1. 对商户来说，“API 请求地址接入说明”和“Sub2API 面板接入说明”是平台分类，不是发布服务的核心决策。
2. 买家提交意向后，本来就需要站外确认接入细节。
3. 有特殊要求的商户可以写在买家须知里。

实现上，如果后端或市场列表仍需要 `deliveryModes`，前端可以保守提交默认值：

```ts
deliveryModes: ['api_key_endpoint']
```

前台展示文案改为：

```text
提交意向后站外确认接入细节
```

如果未来确实要区分面板接入和请求地址，可以放到高级设置或商户备注模板里，而不是主流程。

### 用量与余额查看

建议从主发布表单移除。

理由：

1. 商户主动选择“实时可见”“余额可见”容易制造不可核验承诺。
2. 平台如果没有真实接入面板，就不应该让商户声明平台级可见性。
3. 买家真正需要的是“怎么核对”，这可以写在买家须知或计费说明里。

实现上由系统保守推导：

```ts
if (billingMode === 'fixed_package') usageVisibility = 'fixed_package_only'
else usageVisibility = 'merchant_confirmed'
```

不再让商户选择 `panel_realtime` 或 `panel_balance_only`。如果后续平台有真实集成能力，再由系统或审核后台赋值，而不是商户自填。

## 建议 GPT 重点评审的问题

1. 模型卡片“点击切换”是否比“只能添加，必须在明细表删除”更适合首用者？
2. 唯一已选模型不允许取消，是否会让用户困惑？是否应该允许空状态再用校验提示？
3. 删掉接入方式后，买家侧是否还需要在列表里比较这个维度？如果需要，是否统一展示“站外确认”即可？
4. 删掉用量与余额查看后，是否会损失买家信任信息？还是说保守展示“商户确认用量”更诚实？
5. 展示身份默认隐藏社区身份是否合理？还是应该默认公开个人身份以提升信任？
6. 自动选择第一个模型是否会误导商户？如果保留自动选择，高亮和默认来源提示是否足够？

## 可直接给 GPT 的评审提示词

```text
你是一个产品体验评审和前端交互评审。请评审下面这份“API 服务发布页选中逻辑与表单压缩”方案。

评审重点：
1. 首次发布 API 服务的用户是否能明确知道模型已选中？
2. 模型卡片点击切换、已选模型仍留在列表中，这个逻辑是否合理？
3. 唯一已选模型不允许取消，是否应该改成允许空状态再用校验拦截？
4. 接入方式从主表单移除，统一写成“提交意向后站外确认接入细节”，是否会损害买家比较效率？
5. 用量与余额查看从商户可选项移除，系统默认“商户确认用量”，是否更诚实？
6. 对外展示身份移到提交前 checkbox，是否比放在第一步更符合任务流程？

请按这个格式输出：

- 最高风险：列 1-3 个
- 推荐改法：给出具体交互文案或状态规则
- 可以删除的复杂度：指出哪些字段或说明仍然多余
- 实现注意点：列出前端状态、校验、payload 兼容要点
```

## 验收标准

改完后应满足：

1. 用户点击模型后，当前卡片立即高亮，不发生自动滚动。
2. 不看下方明细表，也能知道哪些模型已选。
3. 发布页主流程少于 5 个大块。
4. 接入方式不再作为商户必填项。
5. 用量与余额查看不再作为商户可声明能力。
6. 提交 payload 仍满足现有后端要求，隐藏字段由前端保守默认或系统推导。
7. 首次进入页面时，有一个可发布的默认草稿，但所有默认值都必须在 UI 上清楚可见。
