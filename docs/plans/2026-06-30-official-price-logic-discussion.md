# 官方价格逻辑讨论稿

日期：2026-06-30
执行者：Codex

## 背景

现在页面里有“官网价格”和“提交低价线索”两条用户路径。用户想看的其实更接近截图里的全球价格排行：选择产品套餐后，看到不同国家或地区的官方原价、折合人民币价格、最低标记和更新时间。

当前系统已经实现了“用户提交低价线索 -> 管理员审核 -> 生成公开价格记录”的完整链路，但这条链路会让公开页同时背负“价格查询”和“众包情报收集”两个产品目标。首发阶段需要重新判断：官方最低价格到底应该由用户发布、管理员手输，还是爬虫采集后确认。

## 现在的逻辑

### 前台入口

- 左侧导航有“官网价格”，进入 `/official-prices`。
- 左侧“发布”分组里有“提交低价线索”，进入 `/official-prices/submit`。
- 官网价格列表页顶部仍显示“提交低价线索”按钮。
- 官网价格详情页有“提交更新线索”入口。

### 用户提交线索

- 用户在 `/official-prices/submit` 填写产品、套餐、地区、渠道、原币价格、开通方式、来源链接和备注。
- 前端调用 `POST /api/v1/official-price-leads`。
- 后端创建 `official_price_leads` 记录，初始状态是 `pending`。
- 用户可以通过 `/api/v1/me/official-price-leads` 查看自己的线索。
- 线索本身不是公开价格事实，只是待审核证据。

### 管理员审核

- 管理员在后台低价线索审核台查看所有线索。
- 管理员可以通过、拒绝、要求复核。
- 审核通过时，管理员需要提供：
  - 解析后的产品套餐 ID。
  - 生效时间。
  - 汇率快照。
  - 审核原因。
- 审核通过后，后端把线索转成 `official_price_records` 公开价格记录。
- 如果同一 offer 已经存在 active 记录，旧记录会被标记为 superseded。

### 公开价格列表

- `GET /api/v1/official-prices` 只返回 `official_price_records.status = active` 的公开记录。
- 列表按 `normalized_monthly_cny ASC` 排序，也就是折合人民币月价从低到高。
- 后端返回 `isLowestReference`，表示这条记录是否是同组里的已验证参考低价。
- 最低参考价分组维度包括：
  - 产品套餐。
  - 地区。
  - 渠道。
  - 开通方式。
  - 计费周期。
  - 价格单位。
  - 是否含税。
- 前端真实后端模式下只展示已验证记录，不展示 pending / rejected / changes_requested 线索。

### 当前问题

- 用户视角上，“提交低价线索”是一个主入口，会让用户以为平台鼓励用户发布价格情报。
- 官方价格页的产品表达仍混有“官网公开价格”和“社区低价线索”两个概念。
- 当前公开价来源依赖“先有用户线索，再由管理员审核通过”，不适合首发阶段快速铺基础价格数据。
- 如果用户提交入口保留在主导航，后续会持续产生审核负担、重复线索和质量问题。
- 现在还没有类似截图那种按产品套餐展示全球地区价格排行的清晰体验。

## 我建议改之后的逻辑

### 产品定位

把“官方最低价”改成“已验证参考低价”或“官方价格参考”。公开页只承载一个目标：帮助用户查某个产品套餐在不同地区、渠道和币种下的公开参考价格。

不再把“发布低价线索”作为首发主功能。用户不应该像发布车源或 API 服务一样发布价格。价格数据应该先由平台维护。

### 数据来源

首发优先使用管理员手输。

管理员在后台维护价格记录，字段包括：

- 产品套餐。
- 国家或地区。
- 渠道。
- 开通方式。
- 原币金额。
- 原币币种。
- 折合人民币价格。
- 汇率来源和汇率时间。
- 来源 URL。
- 观察时间。
- 生效时间。
- 是否含税。
- 状态。

用户线索入口先从主导航和列表页移除。后续如果保留，只作为“纠错/补充来源”入口，挂在价格详情页或空状态里，不作为一级发布能力。

### 公开页

公开页改成更接近截图的价格排行逻辑：

- 顶部按产品和周期切换，例如 Claude Pro 1月、Claude Max 5x 1月、Claude Pro 1年。
- 主体是全球价格表：
  - 排名。
  - 地区。
  - 原价。
  - CNY。
  - 状态。
  - 更新时间。
- 默认按 CNY 从低到高排序。
- 只展示平台确认过的 active 记录。
- 最低标记只给当前产品套餐和周期下 CNY 最低的记录。
- 文案避免承诺“全网最低”，使用“当前已验证最低参考”。

### 后台

后台从“审核用户线索”为主，调整为“维护官方价格记录”为主。

可以保留线索审核模块，但首发不把它暴露给普通用户。后台可以有两个来源：

- 管理员手动新增价格。
- 未来爬虫采集候选价格后，管理员确认入库。

审核动作应该变成“确认价格记录是否可公开”，而不是围绕用户投稿线索展开。

### 爬虫策略

首发不建议直接做自动爬虫上架。

更稳的路径是：

1. 管理员先手输一批高价值产品价格。
2. 后续做半自动采集，爬虫只生成候选记录。
3. 管理员确认候选记录后才进入公开价格表。
4. 自动采集失败、价格异常、汇率异常时，不影响公开数据。

这样可以避免反爬、解析错误、汇率错误和税费口径不一致带来的信任问题。

## 你想要的目标逻辑

你想要的不是“用户发布低价线索”，而是一个像截图一样的官方价格查询工具。

目标用户路径应该是：

1. 用户进入官网价格页。
2. 用户选择产品套餐和周期。
3. 系统展示全球地区价格排行。
4. 用户看到哪个地区最便宜、原价是多少、折合人民币多少。
5. 用户可以点进详情查看来源、更新时间、汇率和说明。
6. 如果用户发现价格不对，可以提交纠错，而不是发布一条公开线索。

目标后台路径应该是：

1. 管理员新增或编辑价格记录。
2. 管理员填写原价、币种、地区、来源和汇率。
3. 系统计算或保存折合人民币价格。
4. 系统按产品套餐和周期自动生成排行。
5. 系统给最低记录打标。
6. 旧价格被新价格替代后，旧记录保留历史但不公开展示。

目标数据来源优先级应该是：

1. 管理员手输确认价格。
2. 爬虫采集候选价格，管理员确认。
3. 用户纠错或补充来源，管理员确认。

## 三种方案对比

| 方案 | 描述 | 优点 | 缺点 | 建议 |
| --- | --- | --- | --- | --- |
| 用户发布低价线索 | 普通用户提交价格，管理员审核后公开 | 有社区参与感，后续数据来源多 | 首发审核负担重，噪音多，容易混淆“线索”和“官方价” | 不作为首发主入口 |
| 管理员手输价格 | 后台直接维护官方参考价格 | 最稳、最快、口径可控 | 初期需要人工整理数据 | 首发推荐 |
| 爬虫采集价格 | 自动抓取外部价格 | 长期可扩展，覆盖面大 | 解析、反爬、汇率、税费、更新频率都复杂 | 后置，先做候选不自动公开 |

## 推荐结论

首发应该改成：

- 前台保留“官网价格”查询页。
- 前台隐藏或移除“提交低价线索”主入口。
- 后台新增或强化“官方价格记录维护”。
- 用户侧只保留“价格纠错/补充来源”入口。
- 数据先由管理员手输。
- 爬虫只作为后续半自动候选来源。

一句话版本：

> 不要让用户发布“官方最低价”，而是让平台维护“已验证参考价”；用户只负责纠错，爬虫只负责提供候选，最终公开数据由管理员确认。

## 需要和 GPT 继续讨论的问题

1. 是否完全删除 `/official-prices/submit`，还是只隐藏入口并保留路由？
2. 后台是复用现有 `official_price_leads -> approve -> records` 链路，还是新增一个直接维护 `official_price_records` 的 admin CRUD？
3. 截图里的产品 tabs 应该来自产品套餐目录，还是价格记录里出现过的产品自动聚合？
4. “最低”标记是按当前 tab 全局最低，还是仍按地区、渠道、开通方式分组最低？
5. 折合 CNY 是管理员手输，还是系统根据汇率快照自动计算？
6. 用户纠错入口是否需要登录？
7. 旧价格历史是否对用户可见，还是只在后台可见？
